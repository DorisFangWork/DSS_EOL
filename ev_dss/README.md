# EV电池 second-life 决策支持系统 (DSS 原型)

一个用于验证"资产退役路径决策"方法论的最小可用原型。
输入退役电池数据,输出每个电池的处置路径推荐(梯次再利用/再制造/转售/回收)、置信度和决策理由。

## 运行

```bash
pip install streamlit pandas numpy pyyaml pulp
cd ev_dss
streamlit run app.py
```

浏览器会自动打开界面。默认加载 `data/batteries.csv`,也可在侧边栏上传自己的 CSV。

## 结构(对应五层架构的第2-5层)

```
ev_dss/
├── app.py              # Streamlit 前端,串起主线
├── config.yaml         # 所有可调参数:归一化边界、权重、路径门槛(你的know-how载体)
├── data/batteries.csv  # 示例输入(第1层"输入适配器"的产物)
└── modules/
    ├── loader.py       # 读取+校验+缺失值处理
    ├── normalize.py    # 第2层:异构读数 → 0-1 归一化
    ├── scoring.py      # 第3层:归一化字段 → 四维评分
    ├── mcda.py         # 第4层:TOPSIS + 路径门槛 → 排序+置信度+理由
    └── optimize.py     # 第4层第二档(可选):批量线性规划优化
```

## 数据流

CSV → loader → normalize → scoring → mcda → 推荐结果 → Streamlit 展示

## 下一步调优方向(这才是真正的工作量所在)

原型代码本身不难,难的是让推荐"准"。真正需要投入的是:

1. **校准 config.yaml**:归一化边界、四维权重、各路径的理想画像和准入门槛——这些数字决定推荐质量,需要用真实历史案例反复调。这是你方法论沉淀的核心。
2. **验证准确率**:找一批已经做过处置决策的真实电池,看系统推荐和实际决策的吻合度。
3. **完善置信度逻辑**:目前是简单的首选-次选差距,可以引入数据质量、边界距离等更细的因素。
4. **接第二个领域**:验证这套引擎换成 BESS/IT 资产时,是否只需改 loader 和 config、不动 mcda——这是通用性的试金石。

## 已知简化(原型阶段有意为之)
